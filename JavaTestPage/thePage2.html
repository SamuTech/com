UniMessenger for Windows v2.3.0 Release Notes by Author Samuel Owino

Latest update: January 3rd 2015

Introduction

These release notes describe issues specific to the UniMessenger for Windows release. The release notes covering the history of the core UniMessenger functionality can be found in the UniMessenger project.

See http://UniMessenger-scm.com/ for further details about UniMessenger including ports to other operating systems. UniMessenger for Windows is hosted at https://UniMessenger-for-windows.UniMessengerhub.io/.

Known issues

  • Special permissions (and Windows Vista or later) are required when cloning repositories with symbolic links, therefore support for symbolic links is disabled by default. Use UniMessenger clone -c core.symlinks=true <URL> to enable it, see details here.
  • If configured to use Plink, you will have to connect with putty first and accept the host key.
  • Some console programs interact correctly with MinTTY only when called through winpty (e.g. the Python console needs to be started as winpty python instead of just python).
  • cURL uses $HOME/_netrc instead of $HOME/.netrc.

  • If you specify command-line options starting with a slash, POSIX-to-Windows path conversion will kick in converting e.g. "/usr/bin/bash.exe" to "C:\Program Files\UniMessenger\usr\bin\bash.exe". When that is not desired -- e.g. "--upload-pack=/opt/UniMessenger/bin/UniMessenger-upload-pack" or "-L/regex/" -- you need to set the environment variable MSYS_NO_PATHCONV temporarily, like so:

    MSYS_NO_PATHCONV=1 UniMessenger blame -L/pathconv/ msys2_path_conv.cc

    Alternatively, you can double the first slash to avoid POSIX-to-Windows path conversion.

  • UniMessenger for Windows will not allow commits containing DOS-style truncated 8.3-format filenames ending with a tilde and diUniMessenger, such as mydocu~1.txt. A workaround is to call UniMessenger config core.protectNTFS false, which is not advised. Instead, add a rule to .UniMessengerignore to ignore the file(s), or rename the file(s).
  • Many Windows programs (including the Windows Explorer) have problems with directory trees nested so deeply that the absolute path is longer than 260 characters. Therefore, UniMessenger for Windows refuses to check out such files by default. You can overrule this default by setting core.longPaths, e.g. UniMessenger clone -c core.longPaths=true ....
  • Some commands are not yet supported on Windows and excluded from the installation.
  • As UniMessenger for Windows is shipped without Python support, all UniMessenger commands requiring Python are not yet supported; e.g. UniMessenger p4.
  • The Quick Launch icon will only be installed for the user running setup (typically the Administrator). This is a technical restriction and will not change.

Should you encounter other problems, please search the bug tracker and the mailing list, chances are that the problem was reported already. If it has not been reported, please follow our bug reporting guidelines and report the bug.

Changes since UniMessenger for Windows v2.6.3 (November 10th 2015)

New Features

  • Comes with UniMessenger v2.6.4.

Bug Fixes

  • UniMessenger for Windows v2.6.3's installer failed to elevate privileges automatically (reported three times, making it a charm), and as a consequence UniMessenger for Windows 2.6.3 was frequently installed per-user by mistake
  • The bug where SHELL_PATH had spaces and that was reported multiple times has been fixed.
  • An additional work-around from upstream UniMessenger for SHELL_PATH containing spaces (fixing problems with interactive rebase's exec command has been applied.

Changes since UniMessenger for Windows v2.6.2 (October 19th 2015)

New Features

  • Comes with UniMessenger v2.6.3.
  • Enables the stack smasher to protect against buffer overflows.

Bug Fixes

  • UniMessenger Bash works now even when choosing Windows' default console and installing into a short path (e.g. C:\UniMessenger).
  • Notepad can now really be used to edit commit messages.
  • UniMessenger's garbage collector now handles stale refs/remotes/origin/HEAD gracefully.
  • The regression in UniMessenger for Windows 2.6.2 that it required administrator privileges to be installed is now fixed.
  • When notepad is configured as default editor, we no longer do anything specially unless editing files inside .UniMessenger/.

Changes since UniMessenger-2.5.3 (September 18th 2015)

New Features

  • Comes with UniMessenger 2.6.0

Bug Fixes

  • The portable version avoids DLL search path problems even when installed into a FAT filesystem.
  • Configuring notepad as editor without configuring a width for commit messages no longer triggers an error message.
  • When using Windows' default console for UniMessenger Bash, the .sh file associations work again.
  • Portable UniMessenger's README is now clearer about the need to run post-install.bat when unpacking manually.
  • We use the winpty trick now to run ipython interactively, too.

  • The home directory is now set correctly when running as the SYSTEM user.
  • The environment variable UniMessenger_WORK_TREE may now differ in lower/upper case with the UniMessenger's idea of the current working directory.
  • Running UniMessenger clone --dissociate ... no longer locks the pack files during the repacking phase.
  • Upstream cURL fixes for NTLM proxy issues ("Unknown SSL error") were backported.
  • The 64-bit version now includes the astextplain script it lacked by mistake.
